# WEB-AGENT Technical Notes

## Реализованная схема работы

1. Агент читает JSON-конфиг.
2. Создаёт рабочие каталоги и файл лога.
3. Проверяет доступность `server_url`.
4. Пытается зарегистрироваться через `wa_reg`.
5. Если регистрация недоступна на стороне сервера, использует `access_code` из конфига.
6. Периодически отправляет `POST` на `wa_task`.
7. При `RUN` разбирает задание и запускает его в отдельном потоке.
8. После завершения отправляет `result_code`, JSON-результат и файлы в `wa_result`.

## Поддерживаемые поля ответа задачи

Агент понимает несколько вариантов названий полей, чтобы быть устойчивым к расхождениям API:

* идентификатор сессии: `session_id`, `session`, `sessionId`
* тип задачи: `task_code`, `task`, `type`
* команда: `command`, `cmd`, `task_cmd`, `task_command`
* команда из API задания: строковое поле `options`
* исполняемый файл: `program`, `executable`, `file`, `path`
* рабочая папка: `working_directory`, `workdir`, `cwd`
* список файлов результата: `result_files`, `files`, `file`, `file1`

## Обработка заданий

* `TASK` возвращает текст stdout/stderr как `message`
* `FILE` без команды отправляет уже готовые файлы из `result_directory`
* `FILE` с командой дополнительно прикладывает найденные файлы из `result_directory`
* если сервер явно перечислил `result_files`, агент ищет именно их
* если список не передан, агент собирает новые файлы, появившиеся в каталоге результатов после старта задачи
* небезопасные пути файлов результата с `..` или абсолютным путём игнорируются

## Надёжность

* логирование потокобезопасно
* при сетевой ошибке интервал опроса увеличивается по backoff до `backoff_max_sec`
* при сбое выполнения задачи агент всё равно пытается отправить ошибку на сервер

## Безопасность

* по умолчанию разрешён только `https://` для `server_url`; `http://` требует явного `allow_insecure_http: true`
* HTTP-редиректы для API-клиента отключены, чтобы исключить утечку `access_code` на сторонние хосты
* проверка TLS-сертификата сервера включена, поддерживаются кастомные `ca_cert_file` / `ca_cert_dir`
* `working_directory` задачи ограничен подкаталогами `result_directory`, выход за пределы отклоняется
* для `result_files` принимаются только безопасные относительные пути без `..` и абсолютных путей
* симлинки в `result_directory`, указывающие наружу, игнорируются при отправке файлов
* на Linux рабочие каталоги агента создаются с правами `0700`, лог-файл — с правами `0600`

## Ограничение текущего стенда

На тестовом URL `https://xdev.arkcom.ru:9999/app/webagent1/` регистрация описана эндпоинтом `wa_reg`; если сервер отвечает, что агент уже зарегистрирован, агент использует заранее выданный `access_code` из конфига.
